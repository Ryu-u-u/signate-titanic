# LightGBM

## このノートブックでやること

XGBoostの「高速版」である LightGBM で、タイタニックの生存予測を行う。

## アルゴリズム解説

### LightGBM = XGBoost の高速版

LightGBM も XGBoost と同じ **勾配ブースティング** だ。
「前の木の間違いを次の木が修正する」という基本原理は同じ。

違いは **木の成長方法** にある。

### Level-wise vs Leaf-wise

```
[XGBoost: Level-wise（レベル単位）]
    すべてのノードを同じ深さまで均等に成長させる
    → バランスの良い木になるが、効率が悪い場合がある

        [root]
       /      \
     [A]      [B]      ← 深さ1: A も B も分割
    /   \    /   \
  [C] [D]  [E]  [F]   ← 深さ2: 全ノードを分割

[LightGBM: Leaf-wise（葉単位）]
    損失を最も減らせる葉だけを優先的に分割する
    → 不均一な形の木になるが、少ない分割で高精度

        [root]
       /      \
     [A]      [B]      ← Aの方が損失削減量が大きい
    /   \       ↑
  [C] [D]     未分割   ← B は後回し
  ↓
 [E]                   ← C をさらに分割（損失削減量が大きければ）
```

**Leaf-wise の利点**: 同じ計算量でより精度が高くなりやすい。
**Leaf-wise の注意**: 深くなりすぎると過学習しやすい → `num_leaves` で制御。

### num_leaves の意味

```
num_leaves = 15
→ 木の葉（末端ノード）の最大数が15個
→ max_depth の代わりに、葉の数で木の複雑さを制御

目安: num_leaves ≤ 2^max_depth
  max_depth=5 なら 2^5=32、num_leaves=15 はかなり控えめ
```

## ハイパーパラメータ

| パラメータ | 値 | 意味 |
|---|---|---|
| `n_estimators` | 500 | 木の本数 |
| `learning_rate` | 0.03 | 学習率 |
| `num_leaves` | 15 | 葉の最大数（木の複雑さを制御） |
| `subsample` | 0.8 | 各木で使うデータの割合 |
| `colsample_bytree` | 0.8 | 各木で使う特徴量の割合 |

## ノートブックの流れ

1. **Setup**: ライブラリ読み込み、設定ファイルの読み込み
2. **Data Loading**: v1特徴量パイプラインで14特徴量を生成
3. **Feature Engineering**: 使用する14特徴量の確認
4. **Cross-Validation**: 5-fold StratifiedKFold で評価
5. **Train Full Model**: 全学習データで再学習、テストデータを予測
6. **Submission**: 提出ファイル生成
7. **Analysis**: 特徴量重要度のバーチャートを可視化

## 実験結果

> **交差検証（CV）って何？** → [01_preprocess/README.md](../01_preprocess/README.md) で詳しく解説している。
> データを5分割し、検証部分を入れ替えながら5回評価→平均する手法だ。

### 5-fold CV 結果

| Fold | Accuracy | F1 | LogLoss | AUC |
|---|---|---|---|---|
| 0 | 0.8090 | 0.7733 | 0.4925 | 0.8643 |
| 1 | 0.8090 | 0.7671 | 0.4665 | 0.8760 |
| 2 | 0.7528 | 0.6765 | 0.6577 | 0.8035 |
| 3 | 0.7753 | 0.7297 | 0.4713 | 0.8661 |
| 4 | 0.7865 | 0.6984 | 0.6033 | 0.8045 |
| **平均** | **0.7865** | **0.7290** | **0.5383** | **0.8429** |

標準偏差（Accuracy）: 0.0238

### なぜ AUC が低いのか？

AUC=0.8429 は他のモデルに比べて低めだが、これは **チューニング不足** の可能性が高い。

- LightGBM は大規模データで真価を発揮する。445件のデータでは過学習しやすい
- `num_leaves=15`, `learning_rate=0.03` は保守的な設定。もう少し調整の余地がある
- LogLoss=0.5383 が高め → 確率の校正（calibration）がうまくいっていない可能性

## ポイント・学び

- **LightGBM は XGBoost と同じ「勾配ブースティング」の仲間**。違いは木の成長方法（Leaf-wise vs Level-wise）
- **大規模データでは LightGBM が XGBoost より高速**。ただし、445件の小さなデータでは差が出にくい
- **num_leaves が LightGBM の最重要パラメータ**。max_depth の代わりに葉の数で複雑さを制御する
- **AUC=0.8429 はチューニング余地あり**。learning_rate, num_leaves, min_child_samples 等を調整すれば改善が見込める
- **小さなデータでの GBDT は過学習に注意**。正則化パラメータの調整が重要だ
