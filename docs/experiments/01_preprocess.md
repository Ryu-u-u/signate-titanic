# 前処理（Preprocessing）

## このノートブックでやること

生データを機械学習モデルに入力できる形に変換する。
欠損値の補完、カテゴリ変数の数値化、派生特徴量の作成を行い、v0（基本版）と v1（拡張版）を比較する。

## 前処理とは何か

ゼロつく1では、MNIST画像を「0〜255の整数」から「0.0〜1.0の浮動小数点」に正規化してからニューラルネットに入力した。
あれが前処理の一種だ。

表形式データでは、さらに多くの処理が必要になる：

```
生データ (CSV)
    ↓ 欠損値補完（NaN → 数値に埋める）
    ↓ カテゴリ変数の数値化（male/female → 0/1）
    ↓ 派生特徴量の作成（既存の列を組み合わせて新しい列を作る）
    ↓ スケーリング（必要なモデルのみ）
学習可能な数値行列
```

## v0 vs v1 の違い

### v0: 基本前処理（9特徴量）

```
元データの列をそのまま数値化するだけ:
  pclass, sex(0/1), age(補完済), sibsp, parch, fare,
  embarked_C, embarked_Q, embarked_S
```

### v1: 拡張前処理（14特徴量）

```
v0の9列 + 5つの派生特徴量:
  family_size    = sibsp + parch + 1     ← 家族全体の人数
  is_alone       = (family_size == 1)    ← 一人旅フラグ
  log_fare       = log(1 + fare)         ← 運賃の対数変換（偏りを緩和）
  fare_per_person = fare / family_size   ← 一人当たり運賃
  pclass_sex     = pclass×10 + sex       ← 客室クラス×性別の交互作用
```

## 交差検証（Cross-Validation）とは何か

ゼロつく1では「訓練データで学習 → テストデータで評価」という1回きりの評価だった。
しかし、たった1回の分割では **たまたまの結果** に左右されてしまう。

**交差検証（CV）** は、データの分割を変えて **複数回** 評価し、平均をとる手法だ。

```
[5-fold Cross-Validation の流れ]

全学習データ (445件) を5等分する

Fold 1: [Val ] [Train] [Train] [Train] [Train] → AUC=0.87
Fold 2: [Train] [Val ] [Train] [Train] [Train] → AUC=0.92
Fold 3: [Train] [Train] [Val ] [Train] [Train] → AUC=0.78
Fold 4: [Train] [Train] [Train] [Val ] [Train] → AUC=0.88
Fold 5: [Train] [Train] [Train] [Train] [Val ] → AUC=0.81

平均 AUC = (0.87 + 0.92 + 0.78 + 0.88 + 0.81) / 5 = 0.852
標準偏差 = 0.050 → ブレの大きさがわかる
```

### なぜ交差検証が必要なのか

1. **偶然の分割に騙されない**: 1回だけの評価では、たまたま簡単なデータが検証に来ただけかもしれない
2. **すべてのデータが1回は検証に使われる**: 445件すべてで予測精度を測れる
3. **安定性がわかる**: 標準偏差が小さければ「どのデータにも安定して効くモデル」、大きければ「ムラがあるモデル」

### StratifiedKFold とは

通常のKFoldでは、ある fold に「生存者ばかり」が偏る可能性がある。
**Stratified**KFold は、各 fold の **生存/死亡の比率を元データと同じに保つ** 分割方法だ。

```
元データの比率: 死亡60% : 生存40%
  → 各foldも約 60:40 に保たれる
  → より公平な評価ができる
```

> この後のモデルREADMEで出てくる「5-fold CV結果」は、すべてこの方法で評価している。

### CV結果の比較

| バージョン | Accuracy | F1 | LogLoss | AUC |
|---|---|---|---|---|
| v0 | 0.7955 | 0.7344 | 0.4606 | 0.8531 |
| v1 | **0.8000** | **0.7410** | **0.4566** | **0.8536** |

AUCの改善は +0.0005 と小さいが、全指標で v1 が上回っている。

## leak-free（データリーク防止）の考え方

**データリーク** = テストデータの情報が学習時に漏れ込むこと。

ゼロつく1では、訓練データとテストデータを最初に分けて、テストデータは最後の評価にしか使わなかった。
同じ原則が前処理にも適用される。

```
やってはいけない例（リーク）:
  全データ(train+test)の平均値で age を補完
  → テストデータの情報が学習データに混入する

正しいやり方:
  学習データの統計量だけで補完値を計算
  → その値をテストデータにも適用する
```

## パイプラインの流れ

```python
# Step 1: 学習データから統計量を計算
train_stats = compute_train_stats(train)
# → age_median_by_pclass: {1: 36.0, 2: 29.0, 3: 25.0}
# → fare_median_by_pclass: {1: 69.3, 2: 15.0, 3: 8.05}
# → embarked_mode: 'S'

# Step 2: 統計量を使って前処理（train, test 両方に同じ統計量を適用）
X_train = build_pipeline(train, version="v1", train_stats=train_stats)
X_test  = build_pipeline(test,  version="v1", train_stats=train_stats)
```

### 欠損値補完の詳細

| カラム | 方法 | 理由 |
|---|---|---|
| age | 客室クラス別の中央値 | EDAでMAR（クラス依存の欠損）と判明 |
| embarked | 最頻値（S） | 欠損2件のみ |
| fare | 客室クラス別の中央値 | テストデータに1件欠損の可能性 |

## ノートブックの流れ

1. **Setup**: ライブラリ読み込み、設定ファイル(`configs/default.yaml`)の読み込み
2. **Data Loading**: train/test読み込み、`compute_train_stats()` で統計量計算
3. **Feature Engineering**: v0 と v1 を `build_pipeline()` で生成、特徴量分布と相関行列を可視化
4. **Cross-Validation**: LogisticRegression で v0 vs v1 の 5-fold CV を実行
5. **Analysis**: fold別のスコアを棒グラフで比較

## ポイント・学び

- **前処理でモデルの性能が変わる**。同じロジスティック回帰でも、特徴量の作り方で AUC が 0.853 → 0.854 と改善した
- **リーク防止は必須**。`compute_train_stats()` → `build_pipeline()` の2段構成で、学習データの統計量だけを使う設計になっている
- **派生特徴量はドメイン知識から生まれる**。「家族サイズ」「一人旅フラグ」はタイタニック号の歴史を知っているからこそ作れる特徴量だ
- **対数変換は偏った分布に効く**。fare の skewness が 3.78 → 0.48 に改善し、モデルが扱いやすくなる
